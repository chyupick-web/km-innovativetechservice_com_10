Starter JS

A typical JS starter will have a number of files. In this article we'll explain what they are and why they're important.

- .vscode
- extensions.json
- launch.json
- settings.json
- .coderoad
- test
- package.json
- .gitignore
- src
- index.js
- .eslintrc.yml
- .gitignore
- package.json
- README.md

#Avoiding Merge Conflicts

Since CodeRoad is built on Git, it's important that we try to avoid merge conflicts. This can be managed in a number of ways:

  • git ignoring files
  • linting & formatting on save

#Git Ignoring Files

First of all, we can ignore meta files that produce common merge conflicts like package-lock.json or yarn.lock.

.gitignore

node_modules
package-lock.json
yarn.lock

#Linting & Formatting on Save

Merge conflicts can be significantly reduces by ensuring consistent & automatic code linting & formatting through tools such as ESLint or Prettier.

While VSCode has its own code formatting defaults, these aren't as strong as the ones provided by ESLint. You can recommend the use of a popular linting extension from the start. Adding files to the .vscode/extensions.json file will prompt the user to install the extension on startup.

.vscode/extensions.json

{
// See http://go.microsoft.com/fwlink/?LinkId=827846
// for documentation on extensions.json
"recommendations": ["dbaeumer.vscode-eslint"]
}

The vscode project can be set to run linting and formatting on save.

See an example below of using Eslint as a linter and formatter for .js files:

.vscode/settings.json

{
// See https://code.visualstudio.com/docs/getstarted/settings
// for documentation on settings.json
// lint on save
"editor.codeActionsOnSave": {
"source.fixAll.eslint": true
},
"eslint.validate": ["javascript"],
// options include ["javascriptreact", "typescript", "typescriptreact", "vue", "html"]
// format on save
"editor.defaultFormatter": "dbaeumer.vscode-eslint",
"eslint.format.enable": true,
"editor.formatOnSave": false, // use eslint formatter
// display eslint "problems"
"eslint.alwaysShowStatus": true
}

Ensure that you avoid eslint best practices that are helpful for production environments but less helpful when learning. These include:

  • no console logs
  • no debugger statements
  • no unused variables

.eslintrc.yml

env:
es6: true
node: true
extends:
- airbnb-base
globals:
Atomics: readonly
SharedArrayBuffer: readonly
parserOptions:
ecmaVersion: 2018
sourceType: module
rules:
# see available rules: https://eslint.org/docs/rules/
no-console: off
no-debugger: off
no-unused-vars: off
